# Builder UX Positioning

Statum generates builders, but Statum is not trying to be a general-purpose
builder crate.

The builder UX goal is narrower:

> Make construction and reconstruction of typed machines predictable, safe, and
> readable.

That keeps builder work tied to Statum's typestate guarantees instead of turning
the project into a broad derive-builder competitor.

## What Statum-Owned Builders Do

### Initial machine builders

Generated by `#[machine]`, these construct a machine in a specific state:

```rust
let draft = DocumentMachine::<Draft>::builder()
    .id("doc-1".to_owned())
    .build();
```

They should optimize for:

- required machine-field coverage,
- compile-time duplicate setter rejection,
- collision checks against generated helper names,
- predictable state-data construction for data-bearing states,
- rust-analyzer-friendly fallback behavior when necessary.

### Rebuild builders

Generated with `#[validators]`, these convert dynamic facts into typed machines:

```rust
let state = DocumentMachine::rebuild(&row)
    .id("doc-1".to_owned())
    .build()?;
```

They should optimize for:

- type-first entry points,
- shared machine context for one row or many rows,
- per-item context with `into_machines_by`,
- diagnostic rebuild reports,
- sync and async validator consistency.

## What Statum Should Not Own

Statum should not try to replace ordinary builder crates for these concerns:

- arbitrary defaults,
- optional-field policy,
- setter renaming for every field,
- `Into<T>` setter customization,
- validation hooks unrelated to machine reconstruction,
- broad derive support for structs that do not represent workflow phases.

Those are good features, but they belong to construction ergonomics rather than
workflow legality.

## Composition Rule

Use an ordinary builder for command/input assembly. Use Statum after the value
enters a lifecycle whose legal method surface changes.

```rust
let command = SubmitDocument::builder()
    .author(author)
    .body(body)
    .build();

let draft = DocumentMachine::<Draft>::from_command(command);
let review = draft.submit_for_review(reviewer);
```

This split is intentional:

- the command builder handles data assembly,
- Statum handles legal workflow phases,
- validators handle dynamic-to-typed rehydration.

## Feature Triage

When a builder feature request appears, classify it before implementing it.

| Request | Statum-owned? | Reason |
| --- | --- | --- |
| Better duplicate setter message | Yes | Directly affects generated typestate builders. |
| Better reserved-name collision message | Yes | Directly affects generated API safety. |
| Per-item machine fields in batch rebuilds | Yes | Typed rehydration boundary. |
| Rebuild reports with validator reasons | Yes | Typed rehydration diagnostics. |
| Optional/default machine fields | Maybe | Useful only if it does not blur required machine context. |
| Setter renaming/custom prefixes | Usually no | Ordinary builder ergonomics. |
| `Into<T>` setters everywhere | Usually no | Broad convenience feature, not lifecycle safety. |
| Generic derive-builder replacement | No | Outside Statum's category. |

## Acceptance Tests For Builder Work

A builder change should usually include at least one of:

- a positive trybuild fixture showing the intended generated API,
- a negative trybuild fixture for a collision or misuse,
- a runtime integration test if it changes rebuild behavior,
- a documentation update if it changes the public construction path.

Run:

```bash
cargo test -p statum-macros
cargo test --workspace --all-features
```
